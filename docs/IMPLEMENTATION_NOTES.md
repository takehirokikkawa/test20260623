# 実装の説明・工夫した点

本書は、TechInsight の実装意図と工夫を、提出要件で指定された観点ごとにまとめたものです。
（UI/UX ／ DB ／ チーム開発 ／ 保守運用・スケーラビリティ ／ その他）

---

## 0. 設計の出発点 — データを見てから設計した

最初に提供データ `articles.csv` を分析し、その特性に設計を合わせました（詳細は [REQUIREMENTS.md](REQUIREMENTS.md) §2）。

- 本文の長さがほぼ均一（平均約312文字）→ **純粋なベクトル類似度だけでは類似度が団子状になり識別力が落ちる**。
- 1,000行中ユニーク本文は約449種類のみ → **埋め込みをハッシュ単位でキャッシュすれば計算量を半分以下にできる**。

この2点が、後述の「ハイブリッド検索」と「埋め込みキャッシュ」という中心的な設計判断の根拠です。
"とりあえずベクトル検索" ではなく、データの性質から逆算して手法を選んでいます。

---

## 1. UI/UX の観点

- **検索のヒット要因を可視化**: セマンティック検索は「なぜこの結果が出たのか」がブラックボックスになりがちです。
  各検索結果に **lexical（語一致）/ fuzzy（あいまい）/ semantic（意味的近接）** のバッジと統合スコアを表示し、
  検索体験の透明性を高めました（要件 FR-8）。
- **状態の明示**: ローディング時はスケルトン、結果ゼロ・エラー時は専用メッセージを表示し、"無言で固まる" UIを排除。
- **楽観的更新（Optimistic UI）**: 作成・編集・削除は即座に一覧へ反映し、API失敗時はロールバック。体感速度を優先。
- **トースト通知**: 操作の成否をトーストで即時フィードバック。
- **モーダル中心の管理UI**: 一覧から離れずに詳細閲覧・投稿・編集・削除が完結。Escキーで閉じられ、ARIA属性で支援技術にも配慮。
- **レスポンシブ**: 1→2→3カラムへ段階的に変化し、モバイルからデスクトップまで対応。

## 2. DB の観点

- **単一DBに検索機能を集約**: 別途ベクトルDBを増やさず、PostgreSQL に `pgvector`・`pg_trgm` を載せて
  ベクトル/全文/あいまい検索を一元化。**運用対象を増やさない**ことを重視しました。
- **インデックスを最初から設計**（[DB_DESIGN.md](DB_DESIGN.md) §3）:
  HNSW（ベクトル近傍）/ GIN（全文検索）/ GIN+trgm（あいまい）/ btree（フィルタ・ソート）を付与し、
  **1万件規模でもフルスキャンを避ける**前提の構造にしています。
- **UUID 主キー + `legacy_id`**: 分散・公開・マージに強いUUIDを採用しつつ、移行元CSVのIDを `legacy_id` に保持して
  トレーサビリティと**冪等な再取り込み**（`ON CONFLICT (legacy_id)`）を両立。
- **生成列 `search_tsv`**: 全文検索ベクトルをDBの GENERATED 列で自動同期。アプリ側の同期漏れを構造的に防止。
- **埋め込みキャッシュ**: 取り込み時に `content_hash`（sha256）で重複本文を判定し、ユニーク本文のみ埋め込み計算。

## 3. チーム開発を意識した観点

- **レイヤード・アーキテクチャ**: backend を `router → service → repository` に分離し、責務境界を明確化。
  検索ロジックは `search_service.py` に集約し、HTTP・永続化と独立させています。
- **契約（コントラクト）駆動**: 先に要件定義・DB設計・API設計を文書化し、それを基準に実装。
  API は Pydantic スキーマ → OpenAPI として公開し、フロントは同じ契約に対して型を定義（FE/BE の型ずれを防止）。
- **テスト容易性**: 検索の核である RRF 統合ロジックを **DB非依存の純粋関数** `reciprocal_rank_fusion` として切り出し、
  単体テスト（`tests/test_search_fusion.py`）で検証可能にしました。スキーマ検証テストも同梱。
- **設定の一元化**: `pydantic-settings` で環境変数を集約。秘匿値はコードに埋め込まず env 経由。
- **ドキュメント整備**: README・要件・DB/API設計・本書を `docs/` に集約し、新規参画者が短時間で全体像を掴める構成。

## 4. 保守運用・スケーラビリティの観点

- **ワンコマンド再現性**: `docker compose up` だけで DB→マイグレーション→データ投入→API→フロントが起動。
  `migrate` を一度きりのジョブに分離し、`depends_on: service_completed_successfully` で起動順序を保証。
- **APIキー不要のローカル埋め込み**: モデルをイメージにビルド時同梱し、実行時は `TRANSFORMERS_OFFLINE=1` で
  ネットワーク非依存。**評価者がキーを持たなくても完全再現**できます（最重要の制約）。
- **スキーマ進化に強い**: Alembic でマイグレーションをバージョン管理。将来の列追加・正規化を安全に適用可能。
- **観測性**: `/health`（DB疎通込み）と構造化ログ、Docker ヘルスチェックを用意。
- **件数増加への備え**:
  - 一覧はページング前提。検索は各手段で候補を上限付き取得 → RRF 統合し、結果集合を小さく保つ。
  - 埋め込み生成はバッチ化済み。作成/更新時の同期生成は、将来バックグラウンドジョブへ非同期化できる設計。
  - 埋め込みモデルは384次元で固定しているため、多言語モデル等へ**列定義を変えずに差し替え可能**。

## 5. その他の観点

- **検索アルゴリズムに RRF を採用**: スコアの絶対値ではなく**順位**で統合するため、本データのように
  ベクトル類似度が上位に密集していても安定して機能します。重み（lexical/vector/fuzzy）は定数化し調整容易。
- **SQLインジェクション対策**: 生SQLは全て `text()` のバインドパラメータ経由で、ユーザー入力を直接連結しません。
- **データ差し替えの容易さ**: `data/articles.csv` を同形式のCSVに置き換えるだけで動作。取り込みは `legacy_id` に対して冪等で、再起動しても重複しません。
- **セキュリティの割り切り（スコープ明示）**: 認証は本課題のスコープ外として未実装ですが、
  レイヤ分離により後付けしやすい構造（依存性注入で認可ミドルウェアを差し込み可能）にしています。

## 既知のトレードオフ / 今後の改善余地

- 認証・認可は未実装（スコープ外）。実運用では管理操作にロールベース認可を追加すべき。
- 作成/更新時の埋め込み生成は同期実行。高頻度更新環境ではジョブキューへ切り出すと応答性が向上。
- author / category は現状 TEXT。マスタテーブルへ正規化すると整合性と集計が強化される。
- 検索の重みやHNSWパラメータ（`ef_search`）は、実データでの評価に基づくチューニング余地がある。
